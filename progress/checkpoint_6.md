## 技术方案

### 生成模块架构（`src/generator/generator.py`）

实现为纯 `src/` 内独立模块，不依赖任何 `scripts/` 文件，可被任意下游代码直接 import。包含以下函数：

**`assemble_context(ranked_chunks, max_context_tokens=6000) → tuple[dict, str]`**

将 reranker 输出的 Top N chunks 去重、按 rerank score 筛选（token 预算内）、按原文阅读顺序排列、格式化为带编号和来源标注的上下文文本。维护 `chunk_1 → f16c_0115` 的编号映射字典，供 JSON 解析时转换。Token 计数使用 `tiktoken` 的 `cl100k_base` encoding。

**`generate_answer(query, context_text, chunk_map) → dict`**

调用 GPT-4o-mini（`temperature=0`，`stream=False`），返回结构化 JSON（`answer` + `used_chunks`）。包含 JSON 解析容错（strip markdown code fence、解析失败时 fallback）。`used_chunks` 中的编号自动转换为真实 chunk_id。

**`extract_sources(ranked_chunks, used_chunk_ids) → list[dict]`**

从 reranker 结果中提取 LLM 实际使用的 chunk 的结构化来源信息（页码、标题路径、图片路径列表）。

**`run_rag_pipeline(query, pdf_name_filter, max_context_tokens) → dict`**

端到端便捷函数，串联检索 → 组装 → 生成的完整流程。

### System Prompt 设计

经过 4 轮迭代测试确定最终版本，共 5 条规则 + JSON 输出格式定义 + 1 条 few-shot 示例：

```
你是 DCS 飞行模拟器手册问答助手。你的唯一知识来源是用户消息中提供的手册片段。

规则：
1. 仅根据提供的片段内容回答。你可以对片段内容进行归纳和总结，但不允许补充片段中完全没有提及的信息。
2. 如果片段中找不到相关信息，但是你的预训练知识中有与问题相关的补充信息（例如解释为什么手册中没有、或提供背景知识），可以在说明片段中无相关信息后补充，但必须明确标注这不来自手册。
3. 如果只能部分回答，据实回答能确认的部分，并说明哪些信息在提供的片段中未涉及。
4. 回答步骤类问题时，保持原文的步骤编号和顺序。
5. 如果用户的描述不够具体但片段中存在可能相关的信息，列出这些信息并建议用户进一步确认。
```

**关键设计决策：**

| 决策 | 最终选择 | 原因 |
|------|---------|------|
| 输出格式 | JSON（answer + used_chunks） | 回答与引用分离，前端可独立消费；used_chunks 实现精确来源标注 |
| 流式 vs 非流式 | 非流式（stream=False） | JSON 输出与流式互斥；GPT-4o-mini 生成速度足够快（2-6s），用户体感可接受 |
| 拒答策略 | 柔性拒答 + 可选预训练补充 | 删除硬性拒答指令后，模型在"该答的答、该拒的拒"之间取得更好平衡 |
| LLM 内嵌页码引用 | 不要求 | 来源信息从 chunk metadata 精确获取，不依赖 LLM 生成页码，避免编造风险 |
| chunk 编号方案 | chunk_1/chunk_2 简单序号 | 真实 chunk_id 对 LLM 无意义，简单编号提高引用准确率 |
| 图片是否发给 LLM | 不发送 | 符合 Text-Anchored 架构设计，成本极低；图片由前端展示 |

### System Prompt 迭代过程

| 轮次 | 主要变化 | 测试结果 |
|------|---------|---------|
| V1 | 6 条规则，含硬性拒答指令"必须回复：手册中未提供相关信息" | 手册外拒答 3/3 ✅，但 q14（黄色警告灯）误拒答 ❌ |
| V2 | 加入规则 5（模糊描述引导），保留硬性拒答 | q14 仍然误拒答 ❌ |
| V3 | 删除硬性拒答指令（规则 2） | q14 正确回答 ✅，手册外拒答 3/3 ✅，整体最佳 |
| V4（最终版） | 恢复规则 2 但改为柔性版本（允许预训练知识补充） | 效果与 V3 等价，保留规则 2 作为未来边缘 case 的许可 |

### Context 组装策略

- **Token 预算**：默认 6000 token 上限，经诊断脚本验证在 15 题基础测试集上未裁剪掉任何目标 chunk
- **排序策略**：先按 rerank score 筛选（保留预算内的 chunk），再按原文阅读顺序排列（`page_start` 升序），确保步骤类回答的顺序正确
- **split_group_id 合并**：已实现但默认关闭，等后续测试决定是否启用

## 主要困难与解决过程

### 1. 硬性拒答指令导致模糊问题误拒答

**问题**：初版 prompt 要求"找不到答案时必须回复'手册中未提供相关信息'"，导致 q14（"黄色警告灯一直亮着"）被误拒答。手册中有大量警告灯信息但没有"黄色"这个词，模型因找不到精确匹配而触发硬性拒答。

**解决**：删除硬性拒答指令，改为柔性表述。模型自然地在"有信息时基于片段回答"和"无信息时坦诚说明"之间取得平衡。规则 1（"不允许补充片段中没有的信息"）单独即可防止幻觉。

### 2. pdf_filter 引号编码不匹配导致检索失败

**问题**：Edge case 测试集首轮运行时全部 12 题拒答，prompt_tokens 仅 280 左右。定位原因为 JSONL 文件中 `pdf_filter` 使用 ASCII 直引号 `"`，而 Qdrant 中存储的 `pdf_name` 使用 Unicode 弯引号 `\u201c` / `\u201d`，过滤不匹配导致检索结果为空。

**解决**：将 `pdf_filter` 值修正为 Unicode 弯引号形式。

### 3. JSON 输出格式的流式兼容性

**问题**：要求 LLM 返回结构化 JSON（answer + used_chunks）与流式输出互斥——流式时前端必须等完整 JSON 返回后才能解析。

**解决**：选择非流式方案。GPT-4o-mini 生成 500 token 的回答约 2-6 秒，加上检索的 1-3 秒，总体 4-8 秒的体感可接受。后续如需流式，可改为"先输出纯文本 + 末尾追加 JSON 引用块"的混合方案。

## Edge Case 失败根因分析

本轮 edge case 测试暴露的 4 个失败 case，**根因均在检索层或 chunking 层**，非生成层问题：

| 失败 Case | 根因 | 对应迭代项 |
|-----------|------|-----------|
| e01（跨章节综合） | 检索只命中语义最近的单一章节，无法同时召回"冷启动"和"滑行"两个章节 | P2 查询分解 |
| e05/e06（精确数值埋在大 chunk） | 冷启动 chunk（3500+ token）包含 EPU 余量和 FLCS BIT 时间等细节，但用户单独问这些细节时，检索命中了更"表面相关"的仪表介绍 chunk | P3 Parent-Child Chunking |
| e07（PRI vs SEC 对比） | 手册中没有专门的对比段落，信息分散在冷启动步骤 24 和发动机章节中，检索无法将分散信息聚合 | P2 查询分解 |

**关键结论**：生成层表现稳定——有 chunk 时回答准确，无 chunk 时坦诚说明，全程无幻觉。系统当前的瓶颈在检索层的召回能力，而非生成层的回答质量。

## 阶段六设计决策（相对原技术方案的调整）

| 原计划 | 实际决策 | 原因 |
|--------|---------|------|
| 流式输出 + Streamlit 前端 | 非流式 JSON 输出，前端推迟至阶段七 | JSON 结构化输出与流式互斥；生成质量验证与前端开发应分离 |
| LLM 在回答中内嵌页码引用 | 不内嵌，来源信息从 metadata 独立提取 | 避免 LLM 编造页码；前端可独立消费来源数据 |
| System Prompt 硬性拒答 | 柔性拒答 + 预训练知识可选补充 | 硬性拒答导致模糊问题误拒答，柔性方案在准确性和用户体验间取得更好平衡 |
| 图片发送给 LLM | 不发送 | 符合 Text-Anchored 架构；单次成本从 $0.001 控制住，发图片会升至 $0.05-0.10 |
| Checkpoint 6 含前端验收 | 仅验收生成质量 | 前端展示推迟至阶段七，本阶段专注于生成质量调优 |

## 已知局限（不影响 MVP）

- **跨章节综合问题**：当答案分散在多个不同章节时，检索倾向于只命中语义最近的单一章节（如 e01）；需查询分解解决
- **精确数值埋在大 chunk 中**：3500+ token 的大 chunk 包含的细节参数（如 EPU 95%-102%、FLCS BIT 45 秒）在用户单独提问时可能未被检索命中
- **预训练知识补充偏保守**：规则 2 允许预训练知识补充，但 GPT-4o-mini 实际上很少主动使用（如 e11 未指出"F-16C 是单座机"），可能需要更强的引导措辞
- **q04（RWS 搜索范围）**：稳定只回答显示距离而漏掉方位角信息，根因是大 chunk 内信息提取不完整
- **图片信息不参与生成**：当答案主要在图片标注中时（如座舱面板布局），纯文字回答可能不够完整
